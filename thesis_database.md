# 知识图谱构建模块

## 数据源设计与多源融合

本系统从多个异构数据源收集技术实体信息，包括维基百科、Stack Overflow、GitHub以及搜索引擎（如DuckDuckGo）。例如，通过爬虫或API自动获取维基百科页面摘要和类别、SO上的问题/答案/代码片段、GitHub中与技术相关的仓库元数据（如star数、依赖信息），并利用搜索引擎补充遗漏信息。由于不同来源数据格式和粒度不同，需要进行融合处理：首先对技术名称进行规范化（如将名称小写并去除特殊字符），以统一多源中的实体标识；然后以此统一实体键将各源信息（摘要、标签、星级等）合并到同一个知识图谱节点上。正如相关研究所述，多源异构数据融合是领域知识图谱构建的核心挑战之一，需要设计针对结构化和非结构化数据的统一处理流程。例如，可以使用Karma、D2R等工具对不同数据源进行抽取并规范化，使得来自维基百科、SO、GitHub的属性能够整合到同一节点内。这种融合方式使得知识图谱能够集成来自多个数据视角的信息，从而获得更全面的技术语义知识。

## 图谱建模与节点设计

知识图谱中的节点代表技术实体或概念。每个节点具有一个或多个标签(label)表示其类别（例如“语言”、“框架”、“工具”、“数据库”等），并包含相关属性(properties)存储元数据。根据Neo4j的模型设计，每个节点都可以有标签标识类型，并可选地携带一组属性。在本系统中，典型的节点属性包括`summary`（技术简介字符串）、`gh_stars`（来自GitHub的星级数值）、`so_tags`（StackOverflow标签列表或数量）等。通过对技术名称进行规范化（如`norm()`函数去除大小写、空格和符号差异），可以确保同一个实体在不同来源中的名称一致，从而避免重复节点。节点的类别可通过额外的标签或属性字段区分，例如给框架类型打上`Framework`标签，给数据库类型打上`Database`标签，或在属性中指定`category`字段。这样，每个技术节点同时包含了丰富的属性数据和语义分类，便于后续查询和可视化。

## 关系设计

节点之间的边表达实体间的语义关联。常见关系类型包括：`USES`关系表示技术依赖（例如一个项目技术栈中的组件使用关系）；`HAS_SO`表示技术实体与StackOverflow问题的关联（即该技术拥有的相关提问）；`HAS_ANSWER`和`HAS_CODE`表示问题与其答案节点或代码片段节点之间的联系；`WIKI_REL`和`SO_REL`则表示通过维基百科或SO标签挖掘出的技术间关联（如维基页面互链或标签共现）。每条关系都有一个类型（如上述名称）和可选的属性，并链接两个节点。在实际构建中，通常使用Cypher的`MERGE`语句创建关系，如`MERGE (a)-[:USES]->(b)`，保证关联边不重复。各关系语义由其名称定义，例如`USES`突出表达依赖使用语义，`HAS_SO`/`HAS_ANSWER`/`HAS_CODE`则明确表示包含关系，帮助捕获技术文档、问答和代码示例之间的结构化联系。总体来看，关系的设计遵循KG的常见模式：以有意义的标签标识类型并连接相应实体。

## 数据写入策略与幂等处理

为了保证多次运行脚本的安全性和幂等性，数据写入使用Neo4j的`MERGE`语句。`MERGE`会尝试匹配已存在的节点或关系，若不存在则自动创建，从而避免重复数据的产生。例如，脚本常用语句`MERGE (t:Tech {name: "...", category: "Framework"})`来保证具有相同名称和类别的技术节点唯一。结合`ON CREATE`和`ON MATCH`子句，可以分别定义首次创建时与已有匹配时的属性赋值逻辑。例如，在`MERGE`后使用`ON CREATE SET t.summary=..., ON MATCH SET t.lastSeen=timestamp()`，即可区分CREATE和UPDATE的处理。通过这种方式，新增节点时填充初始属性，已有节点时仅更新部分字段，实现了安全的插入或更新操作。同时可对关键属性（如名称）建立唯一性约束，进一步确保实体标识的一致性。
